Password Hashing

bcrypt · cost factor 12 · one-way transformation

User passwords are never stored in plain text. When a user is created or their password is set, it is passed through bcrypt — a one-way hashing algorithm specifically designed for passwords.

bcrypt automatically generates a random salt and embeds it into the hash. This means two users with the same password will have completely different hashes in the database.

// Hashing on user creation (seed.js / registration)
const hashed = await bcrypt.hash(password, 12);

// Verification on login
const match = await bcrypt.compare(plainPassword, user.password);
ℹ️
Cost factor 12 means bcrypt performs 2¹² = 4,096 iterations. This makes each hash computation slow enough to resist brute-force attacks, while remaining fast enough for normal login use.
2

AES-256 Encryption

AES-256-CBC · random IV per entry · key stored in .env

All secret text content is encrypted before being saved to MongoDB. The database never contains readable plaintext — only ciphertext. Even if the database were compromised, the data would be unreadable without the encryption key.

The system uses AES-256-CBC (Advanced Encryption Standard, 256-bit key, Cipher Block Chaining mode) via Node.js's built-in crypto module.

🔑 Key Management

The 32-byte AES key is stored in the .env file and loaded via process.env. It never appears in source code or the database.

🎲 Random IV

A fresh 16-byte Initialization Vector (IV) is generated for every encryption. This ensures identical content produces different ciphertext each time.

💾 Storage Format

Stored as ivHex:ciphertextHex — the IV is prepended so decryption can always recover it without a separate field.

🔓 Decryption

Content is only decrypted in memory, on the backend, after the user passes re-authentication. Plaintext is never written to disk or the database.

// encrypt(text) — utils/encryption.js
const iv = crypto.randomBytes(16);
const cipher = crypto.createCipheriv('aes-256-cbc', key, iv);
return iv.toString('hex') + ':' + encrypted;

// What MongoDB stores (example)
"a3f9c1d2e4b5...":"8f2a1c3d4e5f6a7b..."  ← unreadable ciphertext
✅
Defence in depth: Even if an attacker gains direct database access, they cannot read any secret content without the AES key from the server's environment.

JWT Authentication

JSON Web Tokens · 1-hour expiry · Bearer scheme

After a successful login, the server issues a JSON Web Token (JWT) signed with a secret key. This token is sent with every subsequent API request in the Authorization: Bearer <token> header.

The backend verifies the token's signature on every protected route. If the token is missing, tampered with, or expired, the request is rejected with a 401 Unauthorized response.

4

Session Management

Countdown timer · idle timeout · tab-close cleanup

Beyond JWT expiry, Obelisk implements three additional session security mechanisms on the frontend via the useSessionManager hook:

⏱️ Session Countdown

A visible MM:SS timer in the sidebar counts down from 60 minutes. Turns orange at 5 minutes, red at 1 minute. Auto-logs out at zero.

💤 Idle Timeout

If the user has no mouse, keyboard, click, or scroll activity for 2 minutes, they are automatically logged out. Activity resets the timer.

🗂️ Tab Close Cleanup

When the browser tab is closed, the JWT token and user data are removed from localStorage via the pagehide event. Tab switches do not trigger this.

🔔 5-Minute Warning

A toast notification appears when exactly 5 minutes remain in the session, giving the user time to save their work before being logged out.

5

Re-authentication for Sensitive Actions

Password prompt · per-secret trust · 3-minute window

Viewing, editing, or deleting a secret requires the user to re-enter their password, even though they are already logged in. This protects against scenarios where someone gains access to an unlocked screen.

The backend verifies the password via POST /api/auth/verify-password before any sensitive data is decrypted or modified.

Click View / Edit / Delete
→
Password modal opens
→
Backend verifies password
→
Action executes
→
3-min trust granted

Per-secret trusted sessions: After verifying, a 3-minute trusted window is granted for that specific secret's ID. Performing another action on the same secret within 3 minutes skips the prompt. A different secret always requires its own verification.

Decrypted view timer: Once a secret is decrypted and displayed, a 3-minute countdown appears. When it expires, the content is automatically hidden and the trusted session for that secret is revoked — requiring re-authentication to view it again.

ℹ️
Why per-secret? A global trusted session would allow an attacker who gains brief access to view all secrets after one verification. Per-secret trust limits the blast radius to only the secret that was already verified.
6

Login Brute-force Protection

Progressive lockout · escalating durations · stored in DB

The login endpoint tracks failed attempts per user account. After 5 consecutive failures, the account is temporarily locked. Each subsequent lockout doubles the duration.

Lockout #DurationDB Field
1st lockout10 minuteslockCount = 1
2nd lockout20 minuteslockCount = 2
3rd lockout40 minuteslockCount = 3
4th+ lockout60 minutes (cap)lockCount = 4+

The lockUntil timestamp and lockCount are stored in the User document. A successful login resets both fields to zero, forgiving the escalation history.

During a lockout, the error message tells the user how many minutes remain — without revealing whether the account exists (for unknown usernames, a generic "Invalid credentials" is returned).

🚫
Progressive escalation means an attacker who keeps trying after each lockout faces exponentially longer waits — making automated brute-force attacks impractical.
7

Re-auth Rate Limiting

Global attempt counter · 5 failures → 2-min cooldown · all actions blocked

The re-authentication modal also has its own brute-force protection. A global counter tracks failed password attempts across all sensitive actions (View, Edit, Delete).

After 5 global failures, all sensitive actions are blocked for 2 minutes. A visible countdown is shown in the modal. Switching between actions does not reset the counter — it is stored at the module level and shared across all components.

🌐 Global Counter

Stored in a JavaScript module singleton — shared across all hook instances. Switching View → Edit → Delete cannot reset it.

⏱️ Cooldown Display

The modal shows a large MM:SS countdown and disables the password input during the cooldown period.

🎯 Attempt Dots

5 dots in the modal visually show how many attempts have been used. Red dots = failed attempts.

✅ Reset on Success

A successful verification resets the global counter to zero and grants the 3-minute trusted session.

Access Control

JWT middleware · userId scoping · route protection

Every backend route that handles secrets or logs is protected by the verifyToken middleware. Requests without a valid JWT are rejected before reaching any controller logic.

All database queries are scoped to the authenticated user's ID. A user can never read, modify, or delete another user's secrets — even if they know the secret's MongoDB ID.

// Every secret query includes userId — middleware/auth.js
Secret.findOne({ _id: req.params.id, userId: req.user.id })
// If the secret belongs to another user → returns null → 404

The frontend also uses protected routes — unauthenticated users are redirected to the login page before any page renders. The JWT is attached to every API request automatically via an Axios interceptor.

Secure File Handling

Auth-gated downloads · type validation · no public access

Files uploaded with secrets are stored in the server's /uploads directory, which is never exposed as a static public folder. There is no URL that directly serves files.

All file downloads go through an authenticated backend route: GET /api/secrets/:id/download. The middleware verifies the JWT and confirms the file belongs to the requesting user before streaming it.
